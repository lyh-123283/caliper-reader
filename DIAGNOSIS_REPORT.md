## 各阶段问题分级

### 🔴 阶段 0 — 预处理（`preprocess.py`）

**症状**（看 [_debug_out/0_预处理.png](./_debug_out/0_预处理.png) 第3张）：自适应二值图把卡尺**金属面板大面积纹理**都误识别为前景，刻度线淹没在大块灰白噪声里，背景墙也有噪点。

**根本原因**：
1. `gamma=1.5`（压暗高光）让金属反光区压成中灰，反而和刻度线对比变差；本图属于"光照均匀+反光适中"，应该用 `gamma=1.0` 或略微提亮。
2. `adaptive_block_size=31`、`C=5` 在 4096×3072 大图上太小 —— block 只占图宽 0.76%，邻域内全是刻度线，阈值贴着线本身，导致二值结果细碎不连续。
3. `median_ksize=9` 偏大，对细刻度线有损伤（中值核 9 会抹掉宽度 < 4 像素的线）。
4. CLAHE `tile_grid=8×8` 在大图上每个 tile 高达 ~500 px，局部增强失效。

**建议**：
- 预处理参数应**按图像分辨率自适应**：`block_size ≈ max(31, min_dim // 60)`、`median_ksize ≤ 3`。
- gamma 默认改回 1.0。
- CLAHE tile 数按图像缩放：`tile_w = max(8, w // 200)`。

> **❌**：

### 🔴 阶段 1a — ROI 提取（`roi_extract.py`）

**症状**（看 [_debug_out/1a_ROI提取.png](./_debug_out/1a_ROI提取.png)）：ROI 框选区域为 (301, 380)→(1597, 809)，**只截到了主尺中段 3~8 cm 区域 + 游标尺，但漏掉了主尺左半 (0~2)、右半 (8~13)**，最关键的是**游标尺零线对应的主尺刻度"2 cm"被截掉了一部分**。
✏️:主尺部分我们其实不需要那么多，只要游标尺附近的这些刻度就行了，不要求全部，下面的修改意见倒是没问题
**根本原因**：
- COM 质心法基于"投影质心"扩展固定比例宽度 (`x_center_span_ratio=0.30`)：当卡尺只占图宽 ~50% 时，质心会偏向中间黑色压块（那个固定螺丝的活动模块）→ ROI 中心被拉到 ~700px，半宽 ~600px，左边界裁到 100px 处，右边界裁到 1597px（恰好把左侧"0~2 cm"刻度切掉）。
- 而 `x_pad_ratio=0.15` 只在裁切后向外扩 15%，远不够补回切掉的内容。

**建议**：
- 用更鲁棒的"**双向阈值扩展**"取代质心固定宽度：从质心沿左右扫描，遇到 hproj/vproj 显著下降的位置才停。或者基于直方图的 5%~95% 百分位定边界。
- 对于水平刻度卡尺，x 方向应**主动扩展到包含所有等间距刻线**——可以先在质心区域估算 `tick_gap`，再向两侧搜索最后一条仍满足等间距的刻线为止。

### 🔴 阶段 2 — 区域分离（`region_split.py`）

**症状**（看 [_debug_out/2_区域分离.png](./_debug_out/2_区域分离.png)）：分割线（黄色横线）画在了**卡尺顶部压块和主尺刻度之间**（位置太靠上），而真正的主尺/游标分界应该在"主尺刻度行"和"游标尺面板"之间（即 0~9 数字下方那条窄缝）。

**根本原因**：
1. 投影突变法找最深波谷，但顶部"黑色压块顶端"形成了一个更深、更宽的水平投影低谷（黑面板宽度大、像素值低），被优先选为分界。
2. `tick_density_score` 的"上下两侧"判断在这里失效——分界线上方虽然没刻度，但梯度法/闭运算法依然没把它识别为"上无刻度=低分"，因为打分被两侧 `tick_xs` 共有的列覆盖。
3. 物理先验 `fallback_split_ratio=0.60` 也没用到，因为前面方案"成功"了。

**后果**：上半区（"主尺区"）包含了卡尺面板顶部 + 主尺刻度，下半区（"游标区"）也包含了真正的主尺一部分 + 游标尺 + 游标面板 → 后续 zero_x 检测完全错位。

**建议**：
- 给波谷打分时**强约束"上方必须有连续刻线密度"**：上方必须有 ≥ 5 条 tick 且分布在 ≥ 40% 宽度范围内；不满足的低分淘汰。
- 增加**"卡尺压块上沿/下沿"的硬过滤**——黑色压块在水平投影曲线上是一个 ≥ 30 px 的宽低谷，识别并跳过此类宽谷。

### 🔴 阶段 3a — 主尺刻度线检测（`main_scale.py`）

**症状**（看 [_debug_out/3a_主尺刻度线.png](./_debug_out/3a_主尺刻度线.png)）：检测到 **93 条主尺刻线**，肉眼数实际只有约 50 条 (5cm × 10 小刻度)。**严重过检**：把游标尺刻度线、压块边缘、阴影都标成了主尺刻度（图中能看到上方压块螺丝两侧密密麻麻的青色短刻线，全是噪声）。

**根本原因**：
1. 因为区域分离失败（阶段 2），"主尺区"包含了不属于主尺的黑色压块和阴影。
2. `peak_threshold_factor=0.12` 太敏感（"越小越敏感"），把噪声峰也认作刻线。
3. `extract_ticks_from_binary` 的 `threshold = max(col_max * 0.25, 30.0)` 在压块阴影处也能产生足够强的"垂直前景列"被误判为刻线。
4. 等间距补全后 `refine_ticks_by_spacing` 的"网格边缘延伸"反而把右侧主尺超出 ROI 的部分搜出更多伪刻线。

**建议**：
- 阶段 2 修好后这个问题会大幅缓解，但仍需：
  - `peak_threshold_factor` 提高到 0.20 左右（少漏少误）。
  - `extract_ticks` 的列强度阈值应该用 **`col_max * 0.4` + 长度 > 区域高度 25%** 双重过滤（刻度线在主尺区高度方向必然贯穿大半）。
  - `is_long` 判定 `factor=1.3` 太低，导致很多伪刻线被当成长刻线影响 OCR 锚点，建议提到 1.6 左右。

> **审核留空**：

### 🔴 阶段 3b — 主尺 OCR（`ocr.py`）

**症状**（看 [_debug_out/3b_主尺数字OCR.png](./_debug_out/3b_主尺数字OCR.png)）：候选框（蓝色矩形）落在了**卡尺压块和螺丝阴影**上，**完全没框到下面的"0 1 2 3..."数字**。因为没装 OCR 引擎，识别结果全是 `?`，连位置都全错。

**根本原因**：
1. 阶段 2 把分界线画在压块上方，因此"主尺区域"的下半才是真正的主尺；OCR 在该区域内"刻线上方"搜数字，找到的是压块阴影，不是真正数字所在的位置。
2. 未安装 tesseract/easyocr，连退一步识别也无从谈起。
3. `digit_zone='above'` 的假设是"数字在长刻度线上方"，但此卡尺数字（DELIXI 0 1 2 ...）**在长刻度线下方**——这是个**方向常识假设错误**。

**建议**：
- `digit_zone` 应支持自动方向推断：在长刻度线两侧（上+下）都搜连通域，哪侧 OCR 置信度高用哪侧。
- 把 OCR 安装做成"强烈推荐"：`requirements.txt` 里写 `easyocr` 必装，缺失时给明显警告而非沉默退化。
- 修好阶段 2 是治本。

> **审核留空**：

### 🔴 阶段 4a — 游标尺刻度线检测（`vernier_scale.py`）

**症状**（看 [_debug_out/4a_游标刻度线.png](./_debug_out/4a_游标刻度线.png)）：标"ZERO"的蓝线画在了**ROI 最左侧 x=16**（图左上角靠近卡尺主尺 0 cm 的位置），而真正的游标尺零线在图中央"DELIXI"右侧 0 刻度线处。检测到 **109 条游标刻线**，远超物理上 11 条。

**根本原因**：
1. 由于阶段 2 区域分离错误，"游标区"实际上包含了**整条主尺所有刻度线**（数字 3~8 上方那一长条主尺），所以 109 ≈ 主尺刻线数。
2. `_find_zero_tick` 取"最左侧足够长的刻线" → 自然落到主尺最左侧那条线（x≈16），而非真正游标尺的零线。
3. `zero_digit_search_ratio=0.50` 在游标区下半搜数字 "0"，但没装 OCR 引擎，验证全部失败，回退到"最左侧"位置法 → 错。

**建议**：
- 治本：修阶段 2。
- 治标补救：游标尺真正零线的物理特征是 **"该 x 位置下方有一片连续黑色面板"（游标尺金属面板）+ 该刻线长度显著大于邻居**。可以增加"面板检测"：在候选 `zero_x` 下方 30px 内灰度均值是否 < 80（黑面板）。
- 增加合理性检验：`zero_x` 不应小于 `ROI_width * 0.05`（明显贴左边的肯定是误判）。

> **审核留空**：

### 🔴 阶段 4b — 游标对齐（`vernier_scale.py`）

**症状**（看 [_debug_out/4b_游标对齐.png](./_debug_out/4b_游标对齐.png)）：高亮的"对齐刻线"绿圈画在游标尺数字"2~3"附近，**所以读数 0.70 mm 来自 sub_idx*precision = 35*0.02**。但因为零线错了 100 多像素，整个网格全错。

**根本原因**：零线错位 → 所有对齐误差计算全部基准错。

**建议**：
- 零线修好后此步可自然修正。
- 另外 `precision_tick_50=40` 阈值导致 109 条游标线被推断为"0.02 精度"——也是阶段 2 错误的连锁反应。修好后 11 条 → 自动回到 0.1（应 0.05，但已经在合理范围）。

### 🟡 阶段 5 — 读数合并（`merger.py`）

**症状**（看 [_debug_out/99_最终标注.png](./_debug_out/99_最终标注.png) / [_debug_out/5b_读数推导.png](./_debug_out/5b_读数推导.png)）：
- 顶部 banner 显示 "VAR = 0 + 0 = 0.0 mm (fallback)"（主尺纯计数回退也算 0）。
- 因为 `zero_x=16` 在所有主尺刻线（x ≥ 16）的最左边，`extra_ticks=0`，所以主尺 = 0。

**根本原因**：merger 本身逻辑没错，是上游 `zero_x` 错误的连锁结果。

**建议**：
- `_compute_main_reading_with_info` 中可加合理性检测：若 `zero_x` 与第一条主尺刻线差值 < 1 像素，提示"零线异常，请手动指定"。
- 当前 `calc_confidence` **没把"零线置信度"和"OCR 数字置信度"纳入**，导致输出"0.82"假高。建议加入：
  - OCR 数字全失败 → ×0.5
  - 零线没通过 "0" 数字验证 → ×0.7
  - 检测刻线数 / 期望刻线数 偏离 30% → ×0.5

## 📌 修改优先级总结

| 优先级 | 模块 | 问题 | 影响面 |
|---|---|---|---|
| **P0 — 致命** | `region_split.py` | 分界线画在压块上方而非主尺/游标之间 | 后续所有阶段失败 |
| **P0 — 致命** | `roi_extract.py` (1a) | ROI 漏掉主尺左/右两段、把游标尺零线对应区域切掉 | 主尺读数 0 |
| **P0 — 致命** | `vernier_scale.py / _find_zero_tick` | `zero_x` 取到 ROI 最左侧 | 总读数错 |
| **P1 — 配置依赖** | `requirements.txt` + OCR 启动检查 | 未装 OCR 引擎默认沉默退化 | 数字识别全无 |
| **P1 — 鲁棒性** | `preprocess.py` | `block_size`、`tile`、`gamma` 未按图像分辨率自适应 | 二值结果细碎 |
| **P1 — 鲁棒性** | `main_scale.py` | 峰值阈值过敏感、刻线长度过滤不足 | 93 条噪声刻线 |
| **P1 — 方向假设** | `ocr.py / digit_zone` | 强制"数字在上方"假设错误 | 主尺数字框选不到 |
| **P2 — 置信度** | `merger.py / calc_confidence` | 不反映真实失败 | 假高 0.82 |
| **P2 — UI** | `merger.py / 底部面板` | 遮挡游标尺数字 | 仅美观 |

## 🎯 建议修改路线

> **审核此路线**：可改顺序、删除某步、或要求合并某些步骤。

**第 1 步**：先修阶段 2 区域分离，给 `tick_density_score` 加上"上方必须有连续宽度的刻线"强约束 + 物理先验回退（卡尺压块永远在最上方，主尺刻度永远在压块下方的窄长条带）。

**第 2 步**：修阶段 1a ROI，把 x 方向用"等间距刻线搜索"取代固定比例宽度。

**第 3 步**：在 `_find_zero_tick` 加合理性检查（`zero_x` 不能太靠左 + 下方需有金属面板）。

**第 4 步**：把 `easyocr` 提升为强依赖，启动时即时初始化并显示状态。

**第 5 步**：自适应预处理参数（`block_size`、`tile`、`median`）按图像短边缩放。

**第 6 步**：把 OCR 的 `digit_zone` 改为双向自动选择。

**第 7 步**：`merger` 的置信度公式纳入"OCR 失败 / 零线验证失败 / 刻线数偏离"惩罚项。
